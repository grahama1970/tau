# Tau Security Context

`tau.security_context.v1` is the Phase 1 pre-dispatch object for `tau run` and
`tau dag-run`. Tau resolves the DAG goal, policy profile, data boundary,
actor/access manifest, command policy, and environment manifest before any node
command is compiled or dispatched.

## Modes

`--mode development` is available for public or internal work and carries no
production authority.

`--mode secure` requires:

- `policy_profile`
- `data_boundary`
- `actor_access_manifest`
- `command_policy`
- `goal.goal_hash`
- `environment_manifest`, supplied or generated by Tau

Controlled boundaries must use explicit secure mode. Tau blocks before dispatch
when a DAG boundary is `CUI`, `ITAR`, `EAR`, `export_controlled`,
`itar`, or `technical_data`, and the selected mode is omitted or development.

## Path-Backed Inputs

Embedded and path-backed policy/boundary inputs are normalized before gate
requirements are inferred. A relative-path ITAR boundary and an embedded ITAR
boundary must derive the same required gates.

Resolved inputs are hash-bound in `security-context-receipt.json`. The receipt
records whether each input came from an embedded object, relative path, absolute
path, or Tau-generated environment manifest.

## Phase 2.1 Capability Compilation

Secure executable DAG nodes must declare `requested_capabilities`. Tau compiles
those declarations against the resolved command policy after all existing
pre-dispatch gates pass and before command specs are compiled. See
`docs/security-capabilities.md`.

## Secure Executor

Embedded policy profiles and data boundaries are materialized under
`resolved-security-inputs/` so the executor can re-hash the exact resolved
objects before launch. A changed policy or boundary blocks execution.

Secure handoff-loop commands use the grant-bound Bubblewrap executor described
in `docs/secure-executor.md`. Secure ready-queue scheduling remains blocked until
it is routed through the same executor.

## Current Boundary

Secure enforcement currently covers the handoff-loop path used by `tau run` and
`tau dag-run`. Direct lower-level handoff/provider commands remain development
surfaces, and the bounded-ready-queue scheduler is not yet a secure execution
surface.

## Non-Claims

The security-context receipt does not prove ITAR compliance, legal identity,
authoritative U.S.-person status, sandbox enforcement, provider/model semantic
quality, or secure enforcement outside the DAG entrypoints.
